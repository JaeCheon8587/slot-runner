# {App}-TASK-{NNN} — {작업명}

> ⚠ **TEMPLATE** — 모든 `{...}` placeholder를 실제 값으로 채우거나 해당 줄을 삭제한다.
> 본 파일은 단일 App (`{App}`) 의 **AI 실행용 작업 지시서 1건** 이다. 결과 파일명과 문서 ID는 `{App}-TASK-{NNN}` 형식을 사용한다.
>
> **TASK 는 휘발성 + self-contained**: 작업 완료 후 본 파일은 삭제될 수 있다. 따라서 AI 가 코드를 실행하는 데 필요한 컨텍스트 (외부 계약 / 데이터 구조 / 정책 / 코드 변경 단위) 는 본문에 임베드한다. 영구 SSOT (PRD/FC/FRD/ADR/ADR-CATALOG/ARCHITECTURE) 의존을 본문 안에 둘 수 없다.
>
> **외부 SSOT 인용 금지**: 본 TASK 는 영구 SSOT 를 마크다운 링크로 인용하지 않는다 (`[FRD-006](...)` 같은 형태 금지). 영향받는 SSOT 식별은 §6 영향 표에 **이름·요지·갱신 완료 여부만 텍스트로** 명시한다. 역으로 영구 SSOT 도 TASK 를 인용하지 않는다 (휘발성 → 깨진 링크 방지).
>
> **작성 시점 ↔ AI 실행 시점 분리**: 영향 영구 SSOT 갱신은 **작성자가 본 TASK 작성 시점에 동반 수행** 한다 (§6 갱신 완료 여부 = "완료"). AI 가 본 TASK 를 받아 §8 단계를 실행할 때는 **코드 변경만** 수행하며 SSOT 자동 수정 금지.
>
> **작업 유형 메타**: 메타 표의 "작업 유형" 에 분류를 명시한다 — `feature` / `refactor` / `maintenance` / `migration` / `setup` / `investigation` / 기타. **리팩토링 작업도 본 TASK 양식을 사용** (별도 RFD 양식 폐기됨).
>
> **본 TASK 는 코드 검증 기준**: 작업 완료 후 본 문서 ↔ 개발 코드를 `doc-driven-review`(DDR) 로 대조해 정합성(Conformance)을 판정한다. 따라서 §3 목표 상태·§8 작업 단계·§9 완료 기준·§12 컨텍스트 임베드는 **코드로 관찰·대조 가능한 형태**로 작성한다. 특히 코드가 일치해야 할 literal(예외 메시지·출력 포맷·타입/네임스페이스·설정 키)은 §12.2.1 에 verbatim 임베드한다.
