## 2. 범위
| 구분 | 내용 |
|---|---|
| 포함 | 잡 접수(REST)·검증, 영속 세션에 단계 순차 주입, 단계 산출물 파일 게이트, Monday 통지, 완료/실패 팝업·세션 종료/유지 |
| 제외 | 슬롯 풀 다중 동시성·큐(F002), 공용 콘솔(F005), docs-add-task 선행 문서 산출, 봇으로의 결과 역류 |
| 변경되는 사용자 경험 | 운영자가 팝업으로 세션 종료/유지를 결정 |
| 변경되지 않아야 하는 것 | 자동 재시도 없음, LLM 자기보고로 단계 판정하지 않음 |

## 3. 사용자 역할
- 운영자(앱을 켜두고 감독, 팝업 결정)
- Slack 봇(잡 송신 — REST 클라이언트)

## 4. 사전 조건
- 앱이 상시 ON, 빈 슬롯 1개 이상 존재
- 잡 명세에 작업 문서 경로·빌드 대상·대상 식별자(Monday board/item/update)가 포함
- claudecode-for-me 플러그인 슬래시커맨드·Monday MCP 가 세션에서 사용 가능

## 5. 기본 흐름
1. 봇이 잡을 앱 REST 로 전송 → 앱이 즉시 수락 응답
2. 빈 슬롯에 영속 세션 배정·구동
3. 세션에 forge-scope 단계 주입 → 세션이 워크트리에서 구현 → 산출물 파일 기록
4. 앱이 forge 산출물 파일로 성공/차단을 **결정적 판정**(전 step 완료 시에만 다음 단계)
5. 세션에 ddr-loop 단계 주입 → 문서 기준 수렴 검증 → 리뷰 산출물 기록
6. 앱이 ddr 리뷰 산출물 존재를 판정
7. 세션이 Monday 에 결과 댓글 등록 → `Monday`
8. 완료 시 완료 팝업, 실패 시 실패 팝업(사유 포함) 표시
9. 운영자가 팝업에서 세션 종료 또는 유지 선택

## 6. 대안 흐름
- ddr 리뷰 산출물 부재(검토기 부재/무변경): 경고 기록 후 통지 단계로 진행(중단 아님)

## 8. 상세 기능 요구사항
- 단계 전이는 **산출물 파일 판정**으로만 결정한다(LLM 자기보고 비의존) — [ADR-003](../ADR/SLOTRUNNER-ADR-003.md).
- 단계는 정해진 순서(구현 → 문서기준 검증 → 통지)로만 진행. 게이트 미달 시 정지.
- 세션은 작업당 1개로 끝까지 재사용(단계마다 신규 세션 생성 금지) — [ADR-002](../ADR/SLOTRUNNER-ADR-002.md).
- 헤드리스 가정 주입 규칙(무인 실행, 질문/백그라운드 금지)을 운영 프롬프트로 강제.
- 대상 경로·빌드 파라미터는 **프로젝트 논리명(project)** 으로 해석한다 — SlotRunner 호스트 레지스트리가 cwd/sln/app/test_target 을 소유. 봇은 논리명만 보낸다(직접 지정 폴백 허용, 직접값 우선) — [ADR-009](../ADR/SLOTRUNNER-ADR-009.md).
- **루틴(stages)은 봇이 결정해 보낸다**(규칙 기반). SlotRunner 는 받은 stages 를 그대로 실행(재해석 안 함) — [ADR-010](../ADR/SLOTRUNNER-ADR-010.md). 권장 프리셋:

  | 작업 종류 | 입력 신호(봇) | stages |
  |---|---|---|
  | 설계 | `.requirements/…md` / "설계" | `docs-add-task` → `forge-scope` → `ddr-loop` |
  | 개발 | `…/TASK/…md` / "개발" | `forge-scope` → `ddr-loop` |

## 10. 상태 정의
- 파이프라인 단계 상태: 준비 → 구현 → (게이트) → 검증 → (게이트) → 통지 → 완료 | 실패
- 슬롯 상태(요약): 비어있음 → 구동 → 팝업 → 종료 | 유지 (상세 = FRD-002)
- 위 상태는 메모리 보유. 앱 재시작 시 휘발(복구 없음) — [ADR-004](../ADR/SLOTRUNNER-ADR-004.md)
